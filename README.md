# Своя модель перевода fr ⇄ ru для офлайн-работы на телефоне

## Идея пайплайна

Вместо обучения модели с нуля (нереалистично без огромного корпуса и кластера
GPU) — берём уже обученную открытую модель **Helsinki-NLP/opus-mt-fr-ru** /
**opus-mt-ru-fr** (архитектура Marian, ~300 МБ) и:

1. **`scripts/1_prepare_data.py`** — скачивает и чистит параллельный корпус
   fr-ru (OPUS-100). При желании можно дополнительно подмешать Tatoeba —
   там больше "разговорных" коротких примеров, что хорошо подходит именно
   для словарного приложения.
2. **`scripts/2_finetune.py`** — дообучает модель на этих данных
   (можно направить её ближе к вашей предметной области — бытовая лексика,
   короткие фразы — если насобирать данные именно такого типа).
3. **`scripts/3_export_for_device.py`** — конвертирует итоговую модель в
   **CTranslate2** с int8-квантованием. Это уменьшает размер в 3-4 раза
   и ускоряет инференс на CPU в разы — то, что нужно для слабого устройства.

Каждый скрипт запускается по порядку. Шаги 1-2 требуют интернет и
GPU (подойдёт бесплатный Google Colab).

## Род, транскрипция и примеры — без обращения к сайту в рантайме

Дообученная модель перевода не умеет надёжно определять род и придумывать
примеры. Вместо обращения к сайту "на лету" (что требует интернета и
нарушает офлайн-требование), данные берутся из **Викисловаря** — один раз,
на этапе разработки:

1. **`scripts/A_filter_french_wiktionary.py`** — скачайте вручную дамп
   Викисловаря с https://kaikki.org/dictionary/French/kaikki.org-dictionary-French.jsonl
   (544 МБ) и отфильтруйте только французские существительные.
2. **`scripts/B_build_local_db.py`** — соберите из отфильтрованных данных
   компактную базу `local_word_db.json` (слово → род, IPA-транскрипция,
   примеры употребления). Обычно на выходе несколько МБ — это и едет
   вместе с приложением.
3. **`scripts/offline_word_lookup.py`** — итоговый модуль-замена
   `api_worker.py`: переводит слово вашей моделью (шаг 3 предыдущего этапа)
   и достаёт род/транскрипцию/примеры из локальной базы. Если слова нет
   в базе — честно возвращает перевод без рода и без выдуманных примеров,
   а не гадает.

**Почему Викисловарь, а не любой сайт-словарь:** данные Wiktionary
распространяются по лицензии **CC BY-SA / GFDL** — их можно использовать
в своём приложении, нужно лишь указать источник (например, в разделе
"О программе": *"Словарные данные: Wiktionary / kaikki.org"*). Коммерческие
словари (Larousse, Reverso и т.п.) обычно такого не разрешают и запрещают
автоматический парсинг в своих условиях использования.

**Важный нюанс:** структура полей у экспорта Wiktionary периодически
немного меняется. Скрипт `A_filter_french_wiktionary.py` печатает пример
сырой записи при первом запуске — сверьтесь с ним, если `B_build_local_db.py`
соберёт подозрительно мало слов (значит, имена полей отличаются, и функции
`extract_gender`/`extract_ipa`/`extract_examples` нужно чуть поправить).

## Доставка на телефон: реальные варианты

CTranslate2 отлично работает на десктопе и сервере, но для мобильных ОС
(Android/iOS) есть нюансы:

| Вариант | Плюсы | Минусы |
|---|---|---|
| **ONNX Runtime Mobile** (экспорт модели в ONNX) | Официальная поддержка Android/iOS, хорошо документировано | Нужен отдельный экспорт из HF-модели в ONNX (`optimum` от HuggingFace умеет это) |
| **CTranslate2, собранный под ARM** | Максимальная скорость на CPU | Меньше готовых мобильных примеров, придётся собирать C++ под Android NDK самому |
| **Termux / полностью офлайн Python на Android** | Можно почти не менять код | Плохой UX, не то, что ждут от обычного приложения |

## Отдельная проблема: сам интерфейс

Текущее приложение написано на **PyQt6** — десктопный фреймворк, на телефонах
не запускается вообще. Дообученная модель — это только "мозг" перевода,
её ещё нужно встроить в приложение, которое реально ставится на телефон
(Kivy/BeeWare, если хотите остаться на Python, либо нативный
Android/iOS/Flutter-проект).

**Рекомендуемый порядок дальнейших шагов:**
1. Прогнать этот пайплайн на компьютере, оценить качество перевода (BLEU и на глаз)
2. Экспортировать в ONNX через `optimum-cli export onnx`
3. Параллельно решить вопрос с мобильным фреймворком для интерфейса
